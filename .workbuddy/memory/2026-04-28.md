### 技术选型确认

**专家确认：直接使用 sqlx，不使用 tauri-plugin-sql**

原因：tauri-plugin-sql 与 5 项核心设计需求全部不兼容：
1. Executor 泛型模式（不支持）
2. Transaction 管理（需要 `&mut Transaction`）
3. 连接池配置（无法自定义）
4. 错误类型转换（封装了 sqlx::Error）
5. Mock 测试（无法注入）

### 致命缺陷修正

#### 1. 事务模型不可行（P0）
- **问题**：Repository 方法签名 `&SqlitePool` 与事务 `&mut Transaction` 类型不兼容
- **修复**：使用 Executor 泛型模式
  ```rust
  async fn find_by_id<'e, E>(&self, executor: E, id: i64) -> Result<Option<T>>
  where E: Executor<'e, Database = Sqlite> + Send + 'e;
  ```

#### 2. RepositoryError 泄露 sqlx 依赖（P0）
- **问题**：`#[from] sqlx::Error` 使 Service 层与 sqlx 耦合
- **修复**：改为 `Database(String)`，在 SQLite 实现中手动转换
  ```rust
  #[error("数据库错误: {0}")]
  Database(String),  // 不再直接持有 sqlx::Error
  ```

#### 3. Mock 使用阻塞锁（P0）
- **问题**：`parking_lot::Mutex` 会阻塞 tokio 线程
- **修复**：改用 `tokio::sync::Mutex`，所有 `.lock()` 改为 `.lock().await`

#### 4. 迁移元数据表自举问题（P1）
- **问题**：run() 查询 migrations 表，但表可能不存在
- **修复**：添加 `ensure_migration_table()` 在 run() 入口调用

#### 5. 迁移缺少事务包裹（P1）
- **问题**：迁移失败会导致部分执行状态
- **修复**：每个迁移在独立事务中执行

#### 6. AppError 未定义（P2）
- **问题**：Migration trait 使用未定义的 AppError
- **修复**：定义独立的 MigrationError 类型

### 关键技术点

**Executor 泛型模式**：
- Repository 方法接受 `Executor<'e, Database = Sqlite>`
- 支持 `&SqlitePool` 和 `&mut Transaction`
- 事务中可调用 Repository 方法

**RepositoryError 设计**：
- 使用 `Database(String)` 包装错误信息
- Service 层不依赖 sqlx::Error
- 保持 Repository 抽象层的纯粹性

**tokio::sync::Mutex**：
- Mock 实现必须使用异步锁
- 临界区内不允许 `.await`
- 所有 `.lock()` 改为 `.lock().await`

**迁移自举**：
- `ensure_migration_table()` 在 `run()` 入口调用
- 确保 migrations 表存在后再查询
- 解决"表不存在"错误
